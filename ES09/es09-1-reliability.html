Scope

Sources of Faults:
  • Inadequate Specification
  • Design Errors
  • Processor Failure
  • Interference on Communication Channels etc
We will only cover elements of design. We will, however, touch on most of the other sources on our way through.

Definitions

Reliability: How well a system conforms to its specification of behaviour

Failure: A deviation from that specification

Error: The internal problem that results in failure

Fault: The mechanical or algorithmic cause of the error

If a subsystem fails, this can be a fault in the outer system.

Fault → Error → Failure → Fault → Error → Failure …

A failure is a deviation from specification of behaviour which is an external property. If something goes wrong inside a system but its behaviour doesn't change, then it is not a failure. This is important for reliability as it means that we can examine methods to identify and contain faults before they lead to an error and cause a system failure.

Faults

Transient

A fault condition that comes and goes.

Example: Interference on a communication line.

Intermittent

Occur repeatedly but not continuously.

Example: Overheating.

Persistent

Stay in the system until removed by external means.

Example: Damaged component.

Failures

Time-Domain

Result or Action occurs at the wrong time.

Value-Domain

Result or Action is incorrect or inappropriate.

Fail Uncontrolled

Failure is characterised by arbitrary behaviour of the system as a whole.

Fail Soft/Safe

Failure leads to well-defined, safe degredation of performance or shutdown.

Fail Never

Obviates the need for students to sit and listen to talks about reliability.

Fault Prevention

Eliminate possible sources of fault before the system becomes operational.

Fault Tolerance

Provide correct function despite possible faults. Faults are allowed, but isolated so as not to become errors (therefore cannot cause failure.)

Fault Prevention

In order to prevent faults, the system must be understood. What might cause a fault?
  • Understand material properties
  • Know boundary conditions on inputs, outputs
  • Contraints of processor and software systems
  • Real-time performance specifications

Fault Avoidance

Prevent faults from entering the design and constructions.
  • Use high-reliability components
  • Use thoroughly tested methods and techniques for assembly and interconnect
  • Package hardware to prevent external interference
  • Rigorous/Formal specification of requirements and compliance procedures
  • Use proven design methodologies
  • Use standardised software and hardware engineering environments to manage complexity.

Fault Removal

Find and manage faults after construction but before operation.
  • Design Reviews
  • Testing Procedures
  • Program Verification
  • System Testing
  • Hardware Inspection
  • Code Inspection

Production Fault Removal

A subset of fault removal procedures may be used on production units, not just during the design phase.

PCB Test Tag

A portion of the PCB with a special pattern on it. Snapped off the main PCB and saved for future reference if that board should develop a fault.

Other production fault removal procedures may include the isolation of a representative sample of devices from each production run to undergo (potentially destructive) testing in order to ensure conformance to design specifications.

Fault Tolerance

Removal isn't completely effective:

  • Testing can never be exhaustive
  • Test environments are not completely realistic
  • Specification may not be correct
  • Faults may develop at run-time anyway

Fault Tolerance is the concept of providing controlled behaviour in the face of unpredicted fault.

Failures are only defined with respect to the specification. Fault removal must therefore work with respect to the specification. It may not be until the system becomes operational that a deviation between the specification and the real system is detected and in that case, the system may be required to continue to operate (or at least fail safely). Of course there is always the likelihood of faults developing in the field. For example, part damage.

Important For

Systems that are:

  • Remote
  • Unmanned
  • Inaccessible…

Full Tolerance

System continues to function without failure. System parameters (e.g. lifetime) may be affected.

Fail-Soft

Graceful Degredation. The system continues to operate in the face of all “foreseeable” fault conditions, accepting a reduction in functionality or performance.

Fail-Safe

System shuts down safely at fault. Can be thought of as fail-soft degredation to zero functionality.
The level of fault tolerance required will depend on the Embedded System. Many systems will request full fault tolerance but will accept graceful degredation. For example, Rovers on Mars often have a single wheel fail. This subsystem failure is a fault for the rover as a whole with graceful degredation to slower movement.

Tolerance by Redundancy

Redundancy may be used in order to

  • Detect faults early
  • Handle exceptional situations
  • Functional replacements for entire failed (sub-)systems.

We examine redundancy in detail in the next module.

Other Methods

  • Watchdog Timer: Detects when a hardware device hasn't “checked in” for some time and restarts it.
  • Limit Switch: Physically detect mechanical drive issues such as an actuator being instructed to move past its capabilities.
  • Redundant Sensors: Provide sanity checks for each other
  • Transient Recording Systems: “Black Box” that at least allows failure analysis and hopefully prevention of future occurances
